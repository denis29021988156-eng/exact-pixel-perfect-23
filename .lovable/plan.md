## Группировка навигации в сайдбаре

Сейчас 13 пунктов идут одним плоским списком — глаз теряется. Разделим на 4 смысловые группы с подписями:

```
ОБЗОР
  • Сегодня              (брифинг мэра)
  • Карта                (геопространство)

УПРАВЛЕНИЕ
  • Инциденты
  • Программа
  • Поручения
  • Репутация

ИНТЕЛЛЕКТ
  • Шпаргалка            (цифры для доклада)
  • Матрица SLA

ИСТОЧНИКИ ДАННЫХ        ← только для специалистов
  • Качество данных
  • Excel загрузка
  • Telegram
  • AI-структурирование
  • Модерация
```

## Что скрыть от мэра

Мэр не работает с сырыми данными — он видит только результат. Поэтому **вся группа «Источники данных»** (Качество данных, Excel загрузка, Telegram, AI-структурирование, Модерация) скрывается для роли `mayor`. Эти разделы видят только `deputy` и `employee`.

Дополнительно для `mayor` логично скрыть и **«Матрица SLA»** — это операционный инструмент контроля сроков для исполнителей, мэру он не нужен в ЛК.

Итого ЛК мэра = 6 пунктов в 3 группах: Сегодня, Карта / Инциденты, Программа, Поручения, Репутация / Шпаргалка.

## Реализация

В `src/components/AppLayout.tsx`:
- Заменить плоский массив `navItems` на массив групп `[{ label, roles?, items: [...] }]`, где `roles` — опциональный whitelist (если не задан — видно всем).
- Группа «Источники данных» получит `roles: ['deputy','employee']`, пункт «Матрица SLA» — `roles: ['deputy','employee']`.
- В рендере сайдбара выводить заголовок группы (`text-[10px] uppercase tracking-[0.14em] text-sidebar-foreground/40 px-3 pt-5 pb-2`) перед пунктами; на узком (`w-20`) сайдбаре заголовок скрывать, оставив тонкий разделитель.
- Фильтрация по `userRole` из `useAuth()` — пункт/группа просто не рендерится, если роль не подходит.

Никаких изменений маршрутов, БД и логики — только визуальная группировка и условный рендер.
